在 Kubernetes (K8s) 中，**Deployment** 和 **StatefulSet** 是最常用的两种控制器。虽然它们都能管理 Pod，但设计初衷完全不同：一个是为了“无差异化”的扩展，另一个是为了“有身份”的服务。

# 差异对比

| **特性**     | **Deployment (无状态)**  | **StatefulSet (有状态)**              |
| ---------- | --------------------- | ---------------------------------- |
| **Pod 名称** | 随机后缀（如 `web-abcd1`）   | 固定索引（如 `web-0`, `web-1`）           |
| **存储共享**   | 通常多个 Pod 共享同一个 Volume | 每个 Pod 拥有独立的、持久化的 Volume           |
| **启动顺序**   | 并行启动，不分先后             | 顺序启动（0 -> 1 -> 2），上一个 Ready 才启下一个  |
| **网络标识**   | 只有 Service 提供的统一入口    | 拥有唯一的 DNS 域名（Headless Service）     |
| **适用场景**   | Web 前端、微服务、API 网关     | 数据库（MySQL, Redis）、分布式协调（Zookeeper） |

Deployment 认为所有的 Pod 都是**完全相同且可替换的**。

1.  **弹性伸缩：** 如果一个 Pod 挂了，Deployment 会随便再启一个名字不同的 Pod。
2.  **存储：** 如果挂载了存储，所有副本通常看到的是同一份数据（如共享 NFS）。
3.  **更新机制：** 滚动更新时，它会先起新版本的 Pod，等 Ready 后再删旧版本的，过程比较“随意”。

StatefulSet 认为每个 Pod 都是**独一无二的**，必须维持某种“状态”。

1. **持久化身份：** 如果 `web-0` 宕机了，控制器会重新创建一个也叫 `web-0` 的 Pod，并自动挂载回原来的那个磁盘（PVC）。
2. **稳定的网络地址：** 配合 Headless Service（无头服务），每个 Pod 都有固定的内部域名，例如 `web-0.nginx.default.svc.cluster.local`。这对于集群内部节点互相通信（比如分布式选举）至关重要。
3. **VolumeClaimTemplates：** 这是 StatefulSet 的灵魂。它能为每个 Pod 自动创建专属的磁盘，确保 `web-0` 永远用 `data-web-0`。

# 如何选择

### 选 Deployment 的情况：

- 你的应用只是处理请求，不产生本地数据（数据存到远程数据库了）。
- Pod 之间不需要互相知道对方是谁。
- 追求快速部署和极致的水平扩展。

### 选 StatefulSet 的情况：

- **数据一致性：** 比如数据库主从架构，每个节点的数据不一样。
- **集群发现：** 比如 Kafka 或 Zookeeper，节点需要通过固定的 ID 找到彼此。
- **顺序性：** 你的应用必须按照 1、2、3 的顺序启动或关闭。
